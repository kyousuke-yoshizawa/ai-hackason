# AI Harness Core - AIエージェントの基本ルール・制約

対象: Claude Code等、このプロジェクトのコーディングを行うすべてのAIエージェント
このファイルは、エージェントに作業を依頼する際、必ずコンテキストに含めてください。

---

## 1. 絶対に守るルール（MUST）

1. **mainブランチに直接pushしない。** 必ずfeatureブランチを切り、PR経由でマージする。
2. **GitHub Secretsの値を出力・ログに残さない。** APIキーやトークンをコード内にハードコードしない。
3. **`.env` ファイルをコミットしない。** `.gitignore` に含まれていることを都度確認する。
4. **Supabase / Vercelの本番設定を変更する場合は、必ず人間（Kyosuke）に確認を取ってから実行する。**
5. **破壊的な操作（`git push --force`、テーブルのDROP、既存Issueのclose等）は事前に確認を取る。**
6. **不明な要件がある場合は、推測で進めず質問する。**（ハッカソンの時間制約はあるが、誤った実装の手戻りの方がコストが高い）

## 2. 実装方針

- コードは可能な限りシンプルに。20時間という制約の中で「動くこと」を最優先する。
- 過度な抽象化・将来を見越した汎用設計は避ける（ハッカソン後に必要なら別途リファクタ）。
- テストは最低限（重要なロジックのみ）でよいが、`qa-subagent.md` の指示に従い動作確認は必須。
- コミットメッセージは日本語・英語どちらでも可。ただし内容が分かるように具体的に書く。

## 3. コミュニケーションのルール

- 実装完了後は、変更内容をPRの説明欄に簡潔にまとめる（何を・なぜ・どう変えたか）。
- 大きな設計判断（DBスキーマ変更、外部API選定など）は、実装前にIssueコメントで共有する。
- エラーで詰まった場合、30分以上同じ問題で止まっていたら人間にエスカレーションする。

## 4. このプロジェクト固有の制約

- **Supabase / Vercelは1インスタンスを4人で共有** している。他メンバーの作業を壊すスキーマ変更・環境変数の上書きは特に慎重に。
- **Notionへの同期はGitHub Actions経由で自動化されている**（`sync-notion.yml`）。手動でNotionを編集して二重管理にしない。
