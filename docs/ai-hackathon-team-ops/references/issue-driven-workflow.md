# Issue駆動の実装フロー

対象: PM / リード（Kyosuke）、および各自でIssueを立てるメンバー全員

---

## 1. なぜIssue駆動にするか

20時間という短い時間の中で「何を・誰が・いつまでに」やるかを明確にし、
作業の重複や抜け漏れを防ぐため、すべての実装はIssueから始めます。

## 2. Issueテンプレート

```markdown
## タスク概要
（1〜2行で何をするか）

## 背景・目的
（なぜ必要か）

## 完了条件（Definition of Done）
- [ ] 条件1
- [ ] 条件2

## 担当
@ユーザー名

## 見積もり時間
◯時間
```

## 3. ラベル運用

| ラベル | 意味 |
|---|---|
| `frontend` | フロントエンド関連 |
| `backend` | バックエンド関連 |
| `infra` | インフラ・環境関連 |
| `qa` | テスト・QA関連 |
| `priority:high` | 優先度高（先に着手） |
| `blocked` | 他タスク待ちで着手不可 |

## 4. Sprintの分割（20時間の場合の目安）

```
Sprint 1-4（0.5h〜8h）: 基本機能実装
Sprint 5-7（8h〜15h）: 統合・結合テスト
Sprint 8-10（15h〜20h）: 本番化・発表準備
```

各Sprintの開始時にTeamsで「今やるIssue」を宣言し、終了時に進捗を共有します。

## 5. Issue作成〜クローズの流れ

```
1. Kyosukeまたは担当者がIssueを作成
2. 担当者をアサイン（Assignees）
3. 対応するブランチを作成（ブランチ名にIssue番号を含めると追跡しやすい: feature/12-login-form）
4. 実装 → PR作成時に "Closes #12" と書く
5. PRがマージされると自動でIssueがクローズされる
```

## 6. スコープが膨らみそうなとき

ハッカソンでは「あれもこれも」となりがちです。以下の基準で判断してください。

- 完了条件を満たすために **本当に必要か？**
- 20時間以内に収まる見積もりか？
- 収まらなそうなら、Issueを分割し、後回しにできる部分を `priority:low` にする

## 7. Kyosukeの役割（インフラ・PMとして）

- 全体の進捗をIssue一覧（GitHub Projects推奨）で可視化
- ブロッカーになっているIssueを優先的に解消
- Sprint終盤（15h以降）は新規機能追加を止め、安定化に集中させる判断を行う
