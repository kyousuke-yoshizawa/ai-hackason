# QA Subagent - テストガイド（AI向け）

対象: QA担当者、またはテスト実行を依頼されたAIエージェント

---

## 1. 責務範囲

- テストケースの作成
- 手動テストの実行
- バグ報告（Issue化）
- 本番（Vercel）環境での最終確認

## 2. テストケース作成の基本

各主要機能について、最低限以下の観点でテストケースを作る:

- **正常系:** 想定通りの入力で期待通り動くか
- **異常系:** 不正な入力（空文字、極端に長い文字列等）でクラッシュしないか
- **境界値:** 0件・1件・大量データなどの境界

```markdown
## テストケース: ログイン機能
| No | 入力 | 期待結果 |
|---|---|---|
| 1 | 正しいメール/パスワード | ログイン成功、ダッシュボードに遷移 |
| 2 | 間違ったパスワード | エラーメッセージ表示、ログインしない |
| 3 | 空欄のまま送信 | バリデーションエラー表示 |
```

## 3. バグ報告テンプレート（Issue化）

```markdown
## 発生した問題
（何が起きたか）

## 再現手順
1. ...
2. ...
3. ...

## 期待していた動作

## 実際の動作

## 環境
- Preview URL / 本番URL:
- ブラウザ:

## スクリーンショット
（あれば添付）

## 優先度
High / Medium / Low
```

ラベル `qa` を付けてIssue化し、該当の担当者（frontend/backend）をアサインする。

## 4. 本番環境テストのチェックリスト（発表直前に実施）

- [ ] 主要な導線（サインアップ〜メイン機能利用〜結果表示）を一通り実施
- [ ] スマホ幅・PC幅の両方で崩れがないか確認
- [ ] リロードしても状態がおかしくならないか
- [ ] Claude API呼び出しがタイムアウトしないか（レスポンスが遅い場合はローディング表示があるか）
- [ ] エラー時にアプリ全体がクラッシュせず、適切なメッセージが出るか

## 5. デモ用シナリオの準備

発表本番で慌てないよう、以下を事前に用意しておく:

- デモで使うテストアカウント・テストデータ
- うまく動くことが確認済みの操作パス（想定外の操作はデモ中にしない）
- 万が一本番が落ちた場合のバックアッププラン（ローカル/スクリーンショット/動画）

## 6. 優先度判断の目安

- **High:** アプリが動かない、データが消える、セキュリティ上の問題
- **Medium:** 一部機能が期待通り動かないが、回避策がある
- **Low:** 見た目の軽微な崩れ、文言の誤字等（時間があれば対応）
