# Reviewer Subagent - PRレビューガイド（全員向け）

対象: PRレビューを行う全メンバー（役割問わず）

---

## 1. レビューの目的

ハッカソンという時間制約の中でも、「動くか」「壊れないか」「セキュリティ上問題ないか」
の3点は必ず確認する、という共通認識を持つためのガイドです。

## 2. レビュー時に確認する項目

### 必須（Blocking）

- [ ] コードが実際に動く（可能な限りVercel Preview Deployで確認）
- [ ] Secrets/APIキーがハードコードされていない
- [ ] `.env` ファイルがコミットに含まれていない
- [ ] 明らかな無限ループ・無限リクエストなど、Supabase/Vercel/Claude APIの利用量を暴発させるリスクがない

### 推奨（Non-blocking、コメントのみでOK）

- [ ] 命名が分かりやすいか
- [ ] 重複コードが多すぎないか
- [ ] エラーハンドリングが最低限あるか

## 3. コメントの書き方

- 必須修正: `[必須]` を先頭に
- 提案・任意: `[任意]` または `nit:` を先頭に
- 良い実装には積極的に👍やポジティブなコメントを残す

## 4. レビューのスピード感

ハッカソンでは長時間PRを寝かせない。

- レビュー依頼が来たら **30分以内** に一次反応する（すぐに全部見れなくても「見ます」の一言でOK）
- 大きすぎるPR（変更行数が多すぎる）は分割を提案してもよい

## 5. 承認の基準

- 必須項目がすべてクリアされていれば Approve
- 必須項目に問題があれば "Request changes" とし、具体的にどこを直せばよいか明記する

## 6. Kyosuke（最終承認者）が別途確認すること

- ブランチ保護ルール上、レビュー承認後も最終的なマージ判断はKyosukeが行う
- 複数人のレビューで意見が割れた場合、Kyosukeが最終判断する
