要件定義書

架空エリア「ことこと町」対応版 ／ Claude API × 独自店舗データベースで動く、まったり系お出かけプランナー

v1からの主な変更点： 実店舗・実地名（Google Places API）を廃止し、独自に設計した架空エリア・架空店舗を対象にしたシステムへ変更。座標は地図表示専用とし、移動時間は厳密計算せず「距離感タグ＋AI生成」方式に確定。地図ビジュアルも自作SVGで実装する方針を確定。世界観は「どうぶつの森風のまったり可愛いトーン」、エリア名は「ことこと町」に確定。スコアリング重み付け・使用モデル（Claude Sonnet 5）・API呼び出し構成（1回に統合）も確定し、要件定義としては完成状態。

01プロジェクト概要

課題

ユーザーが複数の欲求（「ランチもしたい、映画も見たい、子連れで」など）を抱えるとき、最適な店舗・時間帯を手動で探す必要があり、煩雑である。

ソリューション

LLM（Claude API）で複数欲求を自然言語から解析し、独自に設計した架空エリアの店舗データベースから候補を検索。混雑度・営業時間・回転率を加味した最適ルートプランを複数案提案するAIアシスタント。

架空エリアを採用する理由（v2で追加）

  • 実在店舗・実在ブランド名を扱わないため、商標・実店舗の営業実態との齟齬リスクがゼロになる
  • Google Places APIの料金・レート制限・利用規約を気にする必要がなくなる
  • 混雑度・オファーなど「本来は取得できないデータ」を堂々と自作データとして扱える（v1では「代替策：店舗側入力＋モックデータ」という言い訳が必要だったが、v2ではそれが最初からの設計思想になる）
  • デモの再現性が完全にコントロールできる（実店舗の急な閉店・API障害等に左右されない）

提供価値

  • ユーザー側：複数欲求を1回の入力で解決、「今から行ける」最適プラン、体験品質向上
  • 店舗側：閑散時間への集客、混雑回避、回転率・収益安定化
  • 架空世界という設定により、審査員にもコンセプトそのものを説明しやすい：「これは実際のサービスの縮小版デモであり、本番では実在店舗データに差し替え可能」という位置付け

02世界観・地理設計

v2で新規追加。架空エリアを前提にした、地理表現・地図・店舗マスタの設計をまとめる。

2.1 座標系（方針確定：地図表示用のみ、移動時間は厳密計算しない）

  • 架空エリア全体を XY平面（単位：メートル相当） として定義する
  • 原点(0,0)を基準に、店舗・駅・ランドマークをすべて座標(x, y)で配置する
  • 座標の用途は「地図上への視覚的な配置」に限定する。移動時間をこの座標から数式で厳密算出することはしない

移動時間の生成方式（確定）

  1. 座標から機械的に「距離感タグ」だけを算出する（軽量な閾値判定でよい。近い＝半径150m以内／普通＝150〜400m／遠い＝400m以上）
  2. このタグをClaude APIのプラン生成プロンプトに渡し、「この2店舗は"近い"距離感です。それらしい徒歩移動時間を提案してください」という形でAIに自然な時間を生成させる
  3. 厳密な数式（distance/speedのような計算）は実装しない
設計意図：座標を完全に無視して地図もランダム配置にすると、「地図上は近いのにAIは"徒歩20分"と言う」といった見た目との矛盾が起きやすい。距離感タグという軽いレイヤーだけ挟むことで、矛盾を防ぎつつ「厳密計算はしない」というシンプルさを両立させる。将来バス・電車を追加する場合も同じ枠組みを流用できる。

04データフロー・ユースケース

4.1 ユーザー側：プラン生成フロー

[ユーザー入力] └─ "彼女とランチして、インスタ映えするカフェに行きたい。 ことこと町周辺、2人、予算3000円、15時まで" ↓ [Claude API 呼び出し（1回に統合・確定）] └─ プロンプトに以下をすべて含めて1回で処理する： ① 架空店舗マスタ全8件（カテゴリ・座標由来の距離感タグ・営業時間・価格帯・混雑ルール・オファー） ② スコアリングの重み付け指示（距離感35%・評価25%・混雑度25%・オファー15%） ③ ユーザーの自然言語入力 ④ 出力フォーマット指示（構造化JSON：意図解析結果＋スコア付き候補一覧＋自然文の複数プラン案） ↓ [Claude API 内部処理（1回のレスポンス内で完結）] └─ 意図解析 → 候補抽出（カテゴリ・エリア・予算条件でフィルタ）→ 重み付けスコア計算 → 複数案生成 を1回のレスポンスとして返す 案A: のんびり亭(12:00-13:30) → ことりカフェ(14:00-15:00、徒歩約5分) ★オファー20%OFF 案B: まんまるパンや(12:00-13:00) → きらきらアイス堂(13:30-15:00、徒歩約10分) ↓ [フロントエンド表示] └─ PlanCardコンポーネント（時系列） + SVGマップ（移動経路可視化）
設計メモ：店舗数がデモ規模（8件）と少ないため、「意図解析→DB検索→スコアリング→プラン生成」という4段階すべてを、店舗マスタ全件をコンテキストに含めた1回のClaude API呼び出しでまとめて処理する。これにより直列2回呼び出しによるレイテンシ増を回避し、3秒以内の応答要件を満たしやすくする。店舗数が将来的に増えて全件をコンテキストに含めるのが非効率になった場合は、事前の軽量な絞り込み（キーワード一致等、LLM不使用）を挟む拡張が可能。

4.2 店舗側：設定フロー

[店舗管理画面] ├─ 店舗情報入力（v2で変更） │ └─ 名前・カテゴリ・座標(x,y)・営業時間・価格帯を手動入力 │ （または簡易マップをクリックして座標指定） │ ├─ 混雑パターン設定（曜日×時間帯グリッド） │ └─ 月〜日、0時〜23時の枠で「0=空/1=普通/2=混」を設定 │ ├─ オファー設定 │ └─ 例）"14-16時は狙い目！20%OFF" │ └─ リアルタイム更新ボタン └─ 「今日は混んでる」トグルで即座に反映 ↓ [データベース保存] └─ SQLite: stores, crowd_patterns, offers, area_master テーブルに保存 ↓ [ユーザー側に自動反映] └─ 次のプラン生成時にオファー・混雑度・座標が最新データで計算される

05スコアリング設計（確定）

要素正規化方法ウェイト
距離感タグ近い=1.0 ／ 普通=0.6 ／ 遠い=0.2（座標から機械的に算出するタグを流用）35%
評価（レーティング）0〜5を0〜1に正規化25%
混雑度空=1.0 ／ 普通=0.6 ／ 混=0.225%
オファー有無オファーありなら+0.15加点（別枠）15%

厳密な移動時間の数値そのものはスコアリングに使わず、距離感タグ（3段階）のみを使う。実際にプランに表示する「徒歩◯分」はスコアリングとは別に、Claude APIがタグを見てそれらしく生成する。

07非機能要件

項目要件理由
セキュリティClaude APIのみ外部通信。環境変数でAPIキー管理。ユーザー入力をそのままプロンプトに渡すため、意図解析プロンプトへの注入を軽く意識した設計にする社内ハッカソン安全基準
パフォーマンスプラン生成レスポンス3秒以内。意図解析とプラン生成はClaude API 1回の呼び出しにまとめる（確定）。Tool use / 構造化出力で「構造化JSON（意図解析結果）」と「自然文プラン」を同一レスポンスから取得する設計とし、直列2回呼び出しによるレイテンシ増を回避するユーザーUX
スケーラビリティ店舗数100件程度までのデモ運用を想定。SQLiteで十分ハッカソン最小実装
プライバシーユーザー個人情報の永続保存なし。チャット履歴もセッション内のみ個人情報保護
可用性デモ期間中の継続稼働。外部APIはClaude APIのみになったため、障害時のフォールバックはClaude API呼び出し失敗時の再試行・エラー表示に限定してよいデモ信頼性
世界観の一貫性（新規）架空の店舗名・地名が実在の企業名・商標と偶然一致しないか確認する権利関係の安全性

08制約条件・注意点

  • 解消 Google Maps Popular Timesが取得不可 → そもそも架空データなので自由に設計
  • 解消 Google Places API料金 → 外部API不要
  • ⚠️ Claude API利用料：モデル選定は6.1節で確定済み。チーム共有キーで管理
  • ⚠️ 店舗参加度：架空データなので鶏卵問題自体は発生しないが、「5〜10店舗で世界観として成立しているように見えるか」というリアリティの設計課題に置き換わる
  • ⚠️ デバイス対応：ハッカソン版はブラウザ（PC / タブレット）想定
  • ⚠️ 世界観の命名：実在の駅名・地名・企業名と紛らわしい名称を避ける（商標・誤解リスク回避）

09スコープ・実装優先順

9.1 MVP（最小実装版）- ハッカソン向け

  • 架空エリア・架空店舗データの設計
  • ユーザーチャット入力 → プラン生成表示（3案）
  • 店舗管理画面：基本情報登録（手動）・座標設定・混雑パターン設定・オファー設定
  • 内部DB検索（架空店舗マスタ）
  • Claude API連携（意図解析 + プラン生成）
  • 簡易SVGマップでのプラン可視化
  • SQLite でDB管理

9.2 今回のハッカソンではスコープ外

  • 対象外 ユーザー登録・認証（セッション基盤）
  • 対象外 会話履歴の永続保存
  • 対象外 支払い・予約機能
  • 対象外 スマートフォン完全対応
  • 対象外 複数言語対応
  • 対象外 分析・ビジネスインテリジェンス
  • 対象外 電車・バス等の複数移動手段（徒歩のみ）

9.3 実装順序（クリティカルパス）

  1. 架空エリア・店舗データ設計（エリア名、駅・ランドマーク座標、店舗マスタ8件分をデータとして用意）

  5. 店舗管理API実装 (/api/store, /api/area)
  6. 店舗管理ダッシュボード実装（StorePanel.tsx）

10成功基準・検証

  • 意図解析の精度：複数パターンの自然言語入力で、正確にJSON構造化できるか
  • 内部DB検索の機能性：架空店舗マスタから条件に合う候補が正しく絞り込まれるか
  • 移動時間表現の妥当性：距離感タグ（近い/普通/遠い）とAIが生成する「徒歩◯分」の表現が、地図上の見た目と大きく矛盾しないか
  • プラン生成の自然性：複数案が時系列で矛盾なく生成されるか。移動時間・営業時間が守られているか
  • 店舗側操作の効果：オファー設定・混雑状態更新が即座にプランへ反映されるか
  • 世界観としての説得力：架空エリアであっても、審査員が違和感なく「実在しそうな街」として受け取れるか
  • デモ実演シナリオ：入力→複数案表示のエンドツーエンド動作を確認。複数パターン（ことこと町内の異なるサブエリア等）で実演
