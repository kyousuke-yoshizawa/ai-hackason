## 1. プロジェクト概要

### 課題

ユーザーが複数の欲求（「ランチもしたい、映画も見たい、子連れで」など）を抱えるとき、最適な店舗・時間帯を手動で探す必要があり、煩雑である。

### ソリューション

LLM（Claude API）で複数欲求を自然言語から解析し、**独自に設計した架空エリアの店舗データベース**から候補を検索。混雑度・営業時間・回転率を加味した最適ルートプランを複数案提案するAIアシスタント。

### 架空エリアを採用する理由（v2で追加）

- 実在店舗・実在ブランド名を扱わないため、商標・実店舗の営業実態との齟齬リスクがゼロになる
- Google Places APIの料金・レート制限・利用規約を気にする必要がなくなる
- 混雑度・オファーなど「本来は取得できないデータ」を堂々と自作データとして扱える（v1では「代替策：店舗側入力＋モックデータ」という言い訳が必要だったが、v2ではそれが最初からの設計思想になる）
- デモの再現性が完全にコントロールできる（実店舗の急な閉店・API障害等に左右されない）

### 提供価値

- **ユーザー側：** 複数欲求を1回の入力で解決、「今から行ける」最適プラン、体験品質向上
- **店舗側：** 閑散時間への集客、混雑回避、回転率・収益安定化
- 架空世界という設定により、審査員にもコンセプトそのものを説明しやすい：「これは実際のサービスの縮小版デモであり、本番では実在店舗データに差し替え可能」という位置付け

## 2. 世界観・地理設計

> v2で新規追加。架空エリアを前提にした、地理表現・地図・店舗マスタの設計をまとめる。

### 2.1 座標系（方針確定：地図表示用のみ、移動時間は厳密計算しない）

- 架空エリア全体を **XY平面（単位：メートル相当）** として定義する
- 原点(0,0)を基準に、店舗・駅・ランドマークをすべて座標(x, y)で配置する
- **座標の用途は「地図上への視覚的な配置」に限定する。** 移動時間をこの座標から数式で厳密算出することはしない

#### 移動時間の生成方式（確定）

1. 座標から機械的に「距離感タグ」だけを算出する（軽量な閾値判定でよい。近い＝半径150m以内／普通＝150〜400m／遠い＝400m以上）
2. このタグをClaude APIのプラン生成プロンプトに渡し、「この2店舗は"近い"距離感です。それらしい徒歩移動時間を提案してください」という形でAIに自然な時間を生成させる
3. 厳密な数式（`distance/speed` のような計算）は実装しない

> **設計意図：** 座標を完全に無視して地図もランダム配置にすると、「地図上は近いのにAIは"徒歩20分"と言う」といった見た目との矛盾が起きやすい。距離感タグという軽いレイヤーだけ挟むことで、矛盾を防ぎつつ「厳密計算はしない」というシンプルさを両立させる。将来バス・電車を追加する場合も同じ枠組みを流用できる。

#### 距離感タグの2用途（v2.1で明確化）

距離感タグ（近い／普通／遠い、閾値は近い≤150m／普通≤400m／遠い>400mで両用途共通）は、実装上以下の2つの異なる意味で使われている。要件書上は単一の「距離感タグ」としか書かれていなかったため、区別を明記する。

| 用途 | 基準点 | 使いどころ |
|---|---|---|
| 用途A（スコアリング。5章） | **町中心「どんぐり広場」からの距離**を代理指標として使用 | 店舗単体の「アクセスの良さ」スコア算出（`backend/domains/plan/scoring.ts`） |
| 用途B（プラン生成プロンプト） | **店舗ペアごと**の距離（2店舗間の実座標差） | 訪問順序の決定・移動時間表現の根拠（`buildPairwiseDistanceTable()`、`backend/domains/plan/promptBuilder.ts`） |

> 用途Aの「町中心基準」はスコアリング用の代理指標にすぎず、用途Bの店舗間距離感（プラン内での実際の移動しやすさ）とは別物である点に注意。

## 4. データフロー・ユースケース

### 4.1 ユーザー側：プラン生成フロー

```text
[ユーザー入力]
  └─ "彼女とランチして、インスタ映えするカフェに行きたい。
      ことこと町周辺、2人、予算3000円、15時まで"

  ↓
[Claude API 呼び出し（1回に統合・確定）]
  └─ プロンプトに以下をすべて含めて1回で処理する：
      ① 架空店舗マスタ全8件（カテゴリ・座標由来の距離感タグ・営業時間・価格帯・混雑ルール・オファー）
      ② スコアリングの重み付け指示（距離感35%・評価25%・混雑度25%・オファー15%）
      ③ ユーザーの自然言語入力
      ④ 出力フォーマット指示（構造化JSON：意図解析結果＋スコア付き候補一覧＋自然文の複数プラン案）

  ↓
[Claude API 内部処理（1回のレスポンス内で完結）]
  └─ 意図解析 → 候補抽出（カテゴリ・エリア・予算条件でフィルタ）→ 重み付けスコア計算 → 複数案生成
      を1回のレスポンスとして返す
      案A: のんびり亭(12:00-13:30) → ことりの休憩処(14:00-15:00、徒歩約5分) ★オファー20%OFF
      案B: まんまるパンや(12:00-13:00) → きらきらアイス堂(13:30-15:00、徒歩約10分)

  ↓
[フロントエンド表示]
  └─ PlanCardコンポーネント（時系列） + SVGマップ（移動経路可視化）
```

> **設計メモ：** 店舗数がデモ規模（8件）と少ないため、「意図解析→DB検索→スコアリング→プラン生成」という4段階すべてを、店舗マスタ全件をコンテキストに含めた**1回のClaude API呼び出し**でまとめて処理する。これにより直列2回呼び出しによるレイテンシ増を回避し、3秒以内の応答要件を満たしやすくする。店舗数が将来的に増えて全件をコンテキストに含めるのが非効率になった場合は、事前の軽量な絞り込み（キーワード一致等、LLM不使用）を挟む拡張が可能。

  │
  ├─ 混雑パターン設定（曜日×時間帯グリッド）
  │   └─ 月〜日、0時〜23時の枠で「0=空/1=普通/2=混」を設定
  │
  ├─ オファー設定
  │   └─ 例）"14-16時は狙い目！20%OFF"
  │
  └─ リアルタイム更新ボタン

## 5. スコアリング設計（確定・v2.1で計算式を明文化）

**方式：加重合計（合計85%）＋オファー加点ボーナス（別枠15%）。** 加重平均（合計100%）ではない。オファー無し店舗の理論上限は0.85で、店舗側にオファー設定を促すインセンティブ設計になっている。

| 要素 | 正規化方法 | ウェイト |
|---|---|---|
| 距離感タグ | 近い=1.0 ／ 普通=0.6 ／ 遠い=0.2（座標から機械的に算出するタグを流用） | 35% |
| 評価（レーティング） | **データソース: ユーザーレビュー（`reviews`テーブル）の平均値**（店舗マスタの固定値ではない）。0〜5を0〜1に正規化 | 25% |
| 混雑度 | 空=1.0 ／ 普通=0.6 ／ 混=0.2 | 25% |
| オファー有無 | オファーありなら+0.15加点（別枠。合計85%への加点であり、重み付け平均の1項目ではない） | 15% |

**計算式：**

```
score = min(1, 0.35×距離スコア + 0.25×評価スコア + 0.25×混雑スコア + (オファー有 ? 0.15 : 0))
```

**フォールバック仕様**（`backend/domains/plan/scoring.ts`）：
- レビュー0件（rating=null）の店舗は評価スコアを **0.5**（中間値）として計算する（デモではシードレビュー投入が前提。「レビューシードデータ投入」Issue参照）
- 混雑度が不明（crowdLevel=null）の店舗は混雑スコアを **0.6**（普通相当）として計算する

> 厳密な移動時間の数値そのものはスコアリングに使わず、距離感タグ（3段階）のみを使う。実際にプランに表示する「徒歩◯分」はスコアリングとは別に、Claude APIがタグを見てそれらしく生成する。

## 7. 非機能要件

| 項目 | 要件 | 理由 |
|---|---|---|
| セキュリティ | Claude APIのみ外部通信。環境変数でAPIキー管理。ユーザー入力をそのままプロンプトに渡すため、意図解析プロンプトへの注入を軽く意識した設計にする | 社内ハッカソン安全基準 |

## 10. 成功基準・検証

- **意図解析の精度：** 複数パターンの自然言語入力で、正確にJSON構造化できるか
- **内部DB検索の機能性：** 架空店舗マスタから条件に合う候補が正しく絞り込まれるか
- **移動時間表現の妥当性：** 距離感タグ（近い/普通/遠い）とAIが生成する「徒歩◯分」の表現が、地図上の見た目と大きく矛盾しないか
- **プラン生成の自然性：** 複数案が時系列で矛盾なく生成されるか。移動時間・営業時間が守られているか
- **店舗側操作の効果：** オファー設定・混雑状態更新が即座にプランへ反映されるか
- **世界観としての説得力：** 架空エリアであっても、審査員が違和感なく「実在しそうな街」として受け取れるか
- **デモ実演シナリオ：** 入力→複数案表示のエンドツーエンド動作を確認。複数パターン（ことこと町内の異なるサブエリア等）で実演
